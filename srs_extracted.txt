1. Introduction
1.1 Purpose
This Software Requirements Specification (SRS) defines the complete functional, technical, and operational requirements for Murdock — an AI-powered Indian legal document compiler. It serves as the authoritative reference for all design, development, testing, and deployment decisions. This document supersedes all prior product requirement documents (PRDs).
1.2 Product Overview
Murdock enables Indian citizens, lawyers, businesses, and developers to generate legally structured, court-acceptable, plain-paper documents using AI assistance. Users bring their own AI provider API keys or use platform-managed keys. Documents are generated in English, Hindi, Marathi, and Gujarati. All infrastructure runs within free tiers of Netlify and Supabase to minimise operational cost.
1.3 Scope — v1
Version 1 covers 7 legal modules, 5 user roles, a freemium monetisation model, a full admin dashboard, async AI document generation, per-doc observability, version history for paid users, and multilingual output.

3.3 Request Flow — Document Generation
The following describes the complete end-to-end flow for a user generating a legal document:
User fills intake form (module-specific fields). Auto-suggestions and pre-filled examples are shown.
Client-side pre-processor maps form inputs to a structured JSON schema deterministically (no AI involved yet).
User submits. Netlify Function creates a job row in Supabase (status: queued) and returns a job_id to the browser.
Netlify Background Function picks up the job: fetches user's AI key from Supabase (AES-256 decrypted), calls AI provider with structured JSON + language-aware prompt + module template.
AI returns structured output. Background Function validates output, maps it back to the document schema, and stores result in job row (status: complete).
Browser polls the job status endpoint every 3 seconds ('Generating...' spinner shown). On complete, renders review screen.
Review screen shows document with all AI-filled fields highlighted yellow. User reviews, edits directly, checks mandatory disclaimer checkbox.
User clicks Download → docx.js generates .docx client-side. If paid, document is also saved to Supabase Storage.
⚠ Free tier: Download only. Paid tier: Download + saved to Supabase for future access + version history.
3.4 AI Key Management
AI keys are the most security-sensitive data in the system. The following controls are in place:
Keys stored in Supabase DB encrypted with AES-256. Decrypted only inside Netlify Background Function at call time.
Keys NEVER transmitted to the browser after initial save. Masked display only (last 4 characters shown).
Users can view masked key, view per-provider usage logs, and revoke/rotate key from Settings UI.
Platform-managed keys (owned by Murdock) are available to all users as an alternative to BYOK. These are stored as Netlify Environment Variables — never in the DB.
Platform-managed keys = subscription-based. You (admin) control cost by charging users via the paid tier.

4.2 Pre-Processor — Deterministic Input Mapping
Before any AI call is made, a deterministic pre-processor runs on the user's form submission. This is pure logic — no AI, no randomness. It produces a validated JSON object that is sent to the AI, reducing hallucinations and ensuring consistent outputs.
Each module has a defined JSON schema with required fields, field types, and field-level validation rules.
Critical fields (party names, dates, amounts, addresses, account numbers) are marked as userProvided — AI is instructed never to fill or modify these.
AI-permissible fields (narrative sections, legal boilerplate, cause of action descriptions) are marked as aiGenerate — AI fills only these.
Pre-processor rejects submissions with missing required fields before the job is even queued.
Example pre-processor output (Consumer Complaint):
{ "module": "consumer_complaint", "lang": "en", "complainant": { "name": "[USER_PROVIDED]", "address": "[USER_PROVIDED]" }, "opposite_party": { "name": "[USER_PROVIDED]", "address": "[USER_PROVIDED]" }, "purchase_date": "[USER_PROVIDED]", "purchase_amount": "[USER_PROVIDED]", "defect_description": "[AI_GENERATE]", "relief_sought": "[AI_GENERATE]", "legal_basis": "Consumer Protection Act, 2019" }
4.3 AI Review Flags
All fields populated by AI are highlighted yellow in the review screen. Users must visually inspect every flagged field before downloading. This is a non-skippable step — the download button is locked until the mandatory review checkbox is checked.
Fields filled by user = white background, no flag.
Fields generated by AI = yellow background, labelled 'AI-generated — verify before filing'.
If user edits an AI-generated field on the review screen, the yellow flag is cleared and replaced with a green 'User-verified' indicator.
4.4 Async Job Queue Pattern
Standard Netlify Functions have a 10-second timeout — insufficient for complex AI calls. Netlify Background Functions support up to 15 minutes. The async pattern ensures no timeouts and better UX.
Step 1: Regular Netlify Function receives the request, validates it, creates a job row in Supabase (status: queued), and immediately returns the job_id to the browser (< 1 second).
Step 2: Netlify Background Function is triggered asynchronously. It processes the job (fetch key → call AI → validate output → store result). Updates job row status throughout: queued → processing → complete / failed.
Step 3: Browser polls a lightweight status endpoint every 3 seconds. Shows a 'Generating your document...' spinner with estimated time.
Step 4: On complete, browser fetches the result and renders the review screen. On failed, shows error with retry option.
⚠ Job rows older than 24 hours are automatically purged by a scheduled cleanup function to conserve Supabase DB storage.
5. Legal Modules — v1
All 7 modules produce plain-paper documents acceptable by their respective Indian authority. No stamp paper is required for any module. Documents must be signed by hand after printing. All modules support English, Hindi, Marathi, and Gujarati output.
⚠ Important: Murdock produces correctly structured, legally formatted drafts — not legal advice. The review disclaimer and mandatory checkbox before download make this explicit to users.
5.1 Legal Language & Formatting Basis
Module structure and field schemas are derived from the following authoritative sources, not from assumptions:
Consumer Complaint: Consumer Protection Act 2019 (Sections 35, 36, 58) + NCDRC prescribed proforma and filing guidelines.
RTI Application: RTI Act 2005 (Section 6(1)) + Central Government Form A (CAG Office format) + CIC guidelines. No prescribed format is mandatory — plain paper application with mandatory fields per Section 6(1) is sufficient.
Legal Notice: Code of Civil Procedure 1908 (Section 80 for government, Order XXXVII for commercial) + established Indian legal drafting conventions. Standard sections: parties, cause of action, legal basis, demand, timeline, consequences.
Police Complaint / FIR Draft: Based on standard complaint format used across Indian police stations. No statutory prescribed format — plain paper accepted. Includes all elements required under CrPC Section 154 for cognisable offences.
Employment Grievance Letter: Industrial Disputes Act 1947 (Sections 2A, 33C, 12) + Payment of Wages Act 1936 + POSH Act 2013 (for harassment). Structured for submission to HR, Labour Commissioner, or Labour Court.
Rental Dispute Notice: Transfer of Property Act 1882 + applicable Rent Control Acts (generic structure acceptable across states) + standard tenancy dispute conventions. No stamp required for the notice itself.
Banking / UPI Fraud Complaint: RBI Integrated Ombudsman Scheme 2021 (RB-IOS 2021) fields + National Cyber Crime Reporting Portal (cybercrime.gov.in) complaint fields. Drafted for submission to bank, cybercrime portal (helpline 1930), and RBI Ombudsman if bank fails to resolve within 30 days.

6. Multilingual Output
6.1 Language Strategy
Murdock supports four languages: English, Hindi, Marathi, and Gujarati. Language selection is made at the start of each document generation session. Once selected, the language governs both the document template and the AI prompt — ensuring native phrasing, not translation.
⚠ Critical: Hindi legal terminology != English translated to Hindi. Each language has its own legally appropriate phrasing. AI is instructed to use language-native legal vocabulary, not literal translations.
6.2 Language Implementation

Devanagari
Official language of Maharashtra. Accepted in subordinate courts and administrative offices in Maharashtra. SC has published Marathi-translated judgments. AI uses Marathi legal terminology from Tri-lingual Glossary (EN-Marathi-Hindi) published by Bombay HC.

Gujarati script
Official language of Gujarat. Government of Gujarat sought HC use of Gujarati (pending central approval). Accepted in subordinate courts and all administrative offices in Gujarat. AI prompted to use legal Gujarati vocabulary from Legislative Dept glossary.
⚠ High Courts and Supreme Court proceedings are in English (Article 348). Murdock-generated documents are intended for District Courts, Consumer Commissions, Labour Courts, RTI authorities, police stations, banks, and administrative offices — all of which accept documents in the regional language.

The docx.js generator must embed appropriate fonts for each language to ensure correct rendering on all systems:
English: Arial (universally available)
Hindi: Noto Sans Devanagari or Mangal (widely available on Windows/Android)
Marathi: Noto Sans Devanagari (same script as Hindi, Marathi-specific ligatures supported)
Gujarati: Noto Sans Gujarati or Shruti (Windows built-in)
7. User-Facing Features
7.1 Auto-Suggestions & Pre-filled Examples
To reduce form drop-off, every module includes:
Auto-suggestions: As user types in key fields, contextual suggestions appear. Example: typing 'defective' in the consumer complaint narrative suggests 'defective product', 'defective service', 'defective goods received'.
Pre-filled examples: A 'Load Example' button fills the entire form with a realistic sample case. Example: Consumer Complaint loads a sample mobile phone defect case. User can edit any field. This removes blank-form anxiety.
7.2 Document Review Screen
After AI generation completes, the user sees a formatted document preview with:
Yellow-highlighted fields: All AI-generated text clearly marked for review.
Inline editing: User can click any field and edit directly on the review screen. Edited fields turn green ('User-verified').
Mandatory checkbox: 'I have reviewed all highlighted fields and verified the accuracy of this document. I understand Murdock is not a substitute for legal advice.' Download is locked until checked.
Short disclaimer text: Displayed above the checkbox in smaller font. Non-blocking — user reads it, checks box, proceeds.

All users can manage their AI provider keys from the Settings page:

View key: Only last 4 characters shown (e.g. ...ab7f). Full key never displayed after save.
Usage logs: Per-provider usage log showing date, tokens used, estimated cost, module used, latency.

Platform-managed key: Toggle option available for all users. When enabled, user's own key is not used — platform key is used instead. Charged via subscription tier.
7.6 Free Tier Limits & Enforcement
Free users are limited to 5 document generations per month. This is enforced via per-user usage tracking in Supabase:
Each successful document generation increments the user's monthly doc_count in the users table.
Counter resets on the 1st of each month (UTC).
When limit is reached, generation is blocked and a clear upgrade prompt is shown.
Admin dashboard can override limits for any individual user (e.g. extend free tier, grant unlimited access).

Admin can view all versions for a given module + language combination.
Admin can activate any previous version instantly (rollback).
New version can be drafted and tested in staging before activating in production.

9. Security Requirements
9.1 Authentication & Authorisation
All authentication handled by Clerk. Murdock does not manage passwords or sessions directly.
Every Netlify Function route is protected by Clerk JWT verification middleware.
Role-based access control (RBAC) enforced at the function level — not just the UI.
Admin routes require Admin role claim in the Clerk JWT. Any other role receives 403.
9.2 API Key Security
AI keys encrypted with AES-256-GCM before storage in Supabase. Encryption key stored as Netlify Environment Variable — not in DB.
Key decryption occurs only inside Netlify Background Functions at AI call time. Decrypted key is held in memory only for the duration of the call and never logged.
Keys are never returned to the browser after initial save. Only masked display (last 4 chars).
Platform-managed keys stored as Netlify Environment Variables with restricted access.
9.3 Rate Limiting
All Netlify Function routes are rate-limited using a sliding window counter stored in Supabase:
Unauthenticated routes: 20 requests per 15 minutes per IP.
Authenticated routes: 100 requests per 15 minutes per user.
Document generation route: 10 requests per hour per user (additional guard beyond monthly limit).
Admin routes: 200 requests per 15 minutes per admin user.
Rate limit exceeded: 429 response with Retry-After header.
9.4 Data Security
All Supabase tables use Row Level Security (RLS). Users can only access their own data.
Admin has a separate service role key stored in Netlify Environment Variables — not in the client bundle.
DOCX files in Supabase Storage are stored in private buckets. Accessed only via signed URLs (expiry: 1 hour for personal access, configurable for share links).
No user PII is logged in per-doc logs beyond user_id. No names, no addresses.

Developer sends: module type, language, form fields JSON, and their API key in the request header.
Response: job_id (async). Developer polls status endpoint, retrieves completed document as base64 DOCX.
Developer pays Murdock. Murdock covers the underlying AI provider cost using platform-managed keys for Developer API calls.
Usage logs per developer API key visible to admin.

15. Open Source & Repository Strategy
Repository: GitHub — private from day 1.
6-month review: At 6 months post-launch, owner evaluates whether to make the repo public. Not a commitment — a deliberate decision point.
Developer API: Regardless of open source decision, Developer API access is always a paid feature. Opening the repo does not grant free API access.
Sensitive data: All secrets (Supabase keys, Clerk keys, AI platform keys, AES encryption key) stored as Netlify Environment Variables. Never committed to repo.
Contributions: If repo goes public, a CONTRIBUTING.md and CODE_OF_CONDUCT.md will be published. PRs require maintainer review.
